Most things people build with LLMs are either demos or thin wrappers over API calls. I wanted to build something where the LLM was genuinely doing useful, production-grade work — the kind of thing that actually reduces toil for engineers rather than adding another layer of abstraction they have to babysit.

Runic is an autonomous incident response agent. It detects infrastructure incidents, retrieves relevant context, reasons through hypotheses, plans remediation steps, executes them safely, verifies they worked, and learns from the outcome. The goal wasn't to replace SREs — it was to build something that could handle the routine stuff so humans can focus on the hard problems.

---

## What it actually does

The pipeline is: **Detect → Retrieve → Reason → Act → Verify → Learn**.

An alert comes in from a monitoring system (CloudWatch, Datadog, whatever). Runic normalizes it, scores the anomaly, then pulls context from logs, runbooks, and past incidents using vector search (FAISS + embeddings). That context goes to Gemini, which generates a structured plan — hypotheses, ordered remediation steps with risk levels, rollback strategies, per-step confidence scores.

Then AWS Step Functions orchestrates execution. Low-risk actions run automatically. Medium and high-risk actions require human approval before proceeding. Every action is idempotent, every step has a rollback, and everything is logged — complete audit trail in DynamoDB and CloudWatch.

After execution, it verifies: did the metrics actually return to baseline? If not, it escalates. If yes, it generates an RCA summary and proposes runbook improvements as a GitHub PR.

---

## The tech stack

- **Gemini** for reasoning, planning, and summarization
- **FAISS** for embedding-based retrieval across logs and runbooks
- **AWS Step Functions** for workflow orchestration — this was the right call over rolling my own state machine
- **Lambda** for serverless agent services
- **DynamoDB** for incident lifecycle state
- **S3** for logs, postmortems, and evidence storage
- **AWS CDK (TypeScript)** for infrastructure as code — everything reproducible
- **Cognito** for human-in-the-loop approval flows
- **React + Vite** for an optional incident dashboard

## What went well

The RAG pipeline made a bigger difference than I expected. Without it, the LLM hallucinates remediation steps that don't match the actual infrastructure. With it, the quality of the generated plans improved dramatically — the agent actually references real resource IDs, knows about existing runbooks, and doesn't suggest actions that make no sense for the specific setup.

Step Functions turned out to be exactly the right abstraction for safe execution. Approval gating, retry logic, idempotency, error handling — all built-in. Not having to build a workflow engine from scratch let me focus on the interesting parts: the reasoning pipeline and the tool adapters.

The verification loop (metric-based success checking with statistical thresholds) closes the system in a way that matters. Without it, you have a fancy alert-to-action pipeline that might make things worse. With it, the system has a feedback signal.

---

## What was hard

Getting the confidence scoring right. The LLM is overconfident by default — it will happily assign 0.92 confidence to a plan that would break things. I ended up using a combination of retrieval similarity scores, historical success rates for similar actions, and explicit risk heuristics to calibrate.

Tool adapter design was trickier than I thought. Every AWS service has different error modes, different idempotency guarantees, different rollback semantics. Wrapping them all behind a clean `execute(action) → result` interface took several iterations.

The human-in-the-loop UX is still not where I want it. Right now approval flows go through Cognito + a basic dashboard. The latency between "agent wants to act" and "human approves" can be too long during real incidents. This needs more work.

---

## What I'd do differently

1. **Start with observability, not reasoning.** The LLM is useless without good context. I'd invest more upfront in log aggregation and runbook quality before building the agent.

2. **Test against real incidents earlier.** I spent too long testing with synthetic scenarios. Real incidents are messier — alerts are duplicated, metrics lag, dashboards disagree. The system needs to handle ambiguity gracefully.

3. **Tighter cost control on Gemini calls.** During development, I burned through API credits running the same incidents repeatedly. Caching planning outputs for recurring incident patterns would save a lot.

---

## What's next

Multi-agent collaboration — splitting responsibilities across specialized agents (one for detection, one for diagnosis, one for execution). Reinforcement learning from outcomes so the system genuinely improves over time. Service topology awareness so it understands dependencies when planning remediations.

This project taught me that LLMs are actually useful in production — but only when they're constrained by retrieval, validated by execution feedback, and gated by human approval. The magic isn't the model. It's the system around it.
